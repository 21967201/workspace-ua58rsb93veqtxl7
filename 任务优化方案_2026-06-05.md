### 方案3：添加任务监控

#### 监控1: 任务执行失败监控

**监控目标**: 任务执行失败超过3次 → 发送告警到微信

**实现方案**:
1. 创建监控脚本：`D:\QClawX\data\workspace-ua58rsb93veqtxl7\task_monitor.py`
2. 脚本功能：
   - 读取所有任务的 `state.consecutiveErrors`
   - 如果 `consecutiveErrors >= 3`，发送告警
3. 添加到 cron 任务，每天 20:00 执行

**告警内容**:
```
【任务执行失败告警】

任务名称：<任务名称>
失败次数：<consecutiveErrors>
最后失败时间：<lastRunAtMs>
最后错误信息：<lastError>

请检查任务配置和执行日志。
```

#### 监控2: 任务执行时间监控

**监控目标**: 任务执行时间超过预期 → 发送告警到微信

**实现方案**:
1. 在 `task_monitor.py` 中添加执行时间监控
2. 预期执行时间（根据历史数据）：
   - 1688全景分析：60秒
   - GBrain全景管理：50秒
   - 价格全景监控：30秒

3. 如果实际执行时间 > 预期执行时间 × 2，发送告警

**告警内容**:
```

任务名称：<任务名称>

超时倍数：<实际时间 / 预期时间>

请检查任务执行日志和系统资源。
```

#### 监控3: 任务执行结果异常监控

**监控目标**: 任务执行结果异常 → 发送告警到微信

**实现方案**:

## 📝 风险与注意事项

### 风险1: 任务合并后执行时间变长

**风险描述**: 合并后的任务执行时间可能超过预期，导致任务超时

**应对措施**:
- 在任务 message 中添加超时处理
- 监控合并后任务的实际执行时间
- 如果超时，调整任务配置或拆分任务

### 风险2: 任务合并后失败影响范围变大

**风险描述**: 合并后的任务如果失败，影响范围比原来大

**应对措施**:
- 在任务 message 中添加错误处理
- 监控合并后任务的执行状态
- 如果失败，快速回滚到合并前状态

### 风险3: 任务时间调整可能影响业务

**风险描述**: 调整任务时间可能影响业务数据时效性

**应对措施**:
- 提前通知相关业务方
- 在业务低峰期调整任务时间
- 监控调整后任务执行结果
